STRUCTURAL DECISIONS
---
- Departed from standard 14-section core doc template; used a purpose-built 14-section structure appropriate for a review and quality-control document
  Rationale: standard sections (e.g., "State / Workflow / Controls") do not map to a checklist document; the review categories, check items, and outcome definitions require their own structure

- Three-tier reference model used in Section 5 (Governance Anchors / Foundation Reference Documents / Process Standards)
  Rationale: cleanest way to explain why some check failures are automatic FAILs (Tier 1) while others are flags that need revision (Tier 2) and others are process corrections (Tier 3)

- Mandatory checks written as yes/no questions with YES = pass
  Rationale: concrete enough to actually use during a review; not open to interpretation; consistent with how functional checklists work in regulated environments

- Conditional checks grouped by subject matter (investor eligibility / data storage / transfer restriction / white-label / reconciliation / TA integration / cross-border)
  Rationale: not every document discusses every topic; reviewers should apply only the relevant sub-sections; this avoids "not applicable" filling up review records

- Review outcomes kept to 5 codes — not a sliding scale
  Rationale: outcomes must be clearly differentiated; too many gradations create ambiguity about when a document is actually safe to use

REVIEW OUTCOME MODEL
---
PASS / READY: all mandatory checks pass; safe to advance to READY or hand downstream
PASS / OPEN ITEMS: passes; non-blocking open items correctly handled; stays at DRAFTED
PARTIAL / NEEDS REVISION: conditional failures; no Tier 1 contradiction; revision needed before handoff
BLOCKED / NEEDS INPUT: mandatory check failure requiring external input; advance blocked
FAIL / CONTRADICTION: Tier 1 failure; contradicts locked decision; must be corrected before any other step

Note: "PASS / READY" and "PASS / OPEN ITEMS" are differentiated because a document with open items may be safe to use downstream (Phase 2 authors can acknowledge the open items) but is not yet READY for external or legal review.

ESCALATION MODEL
---
7 escalation types defined:
1. Legal review needed — [REQUIRES REG D COUNSEL INPUT] / [REQUIRES CROSS-BORDER LEGAL INPUT]
2. Vendor confirmation needed — [REQUIRES SECURITIZE CONFIRMATION]
3. Internal decision needed — [REQUIRES INTERNAL DECISION] / [REQUIRES SECOND STREET INPUT]
4. Terminology / glossary conflict — author corrects; new term requires glossary addition before use
5. Architectural contradiction — project owner review required before silent Phase 3 change
6. Scope inflation — out-of-scope; requires documented scope change
7. Completion-layer collapse — Tier 1-adjacent; return to author immediately

Completion-layer collapse (type 7) is treated with the same urgency as a Tier 1 failure even though it is a process failure rather than a direct locked-decision contradiction. Reason: collapsing CHAIN_EXECUTED / TA_RECORDED / LEGALLY_COMPLETE is the single most dangerous documentation error in this domain and could affect investor-facing legal representations.

ASSUMPTIONS USED
---
- "Implementation-Readiness Discipline" (CAT-12) is treated as a process-tier check, not a foundation-tier check. Rationale: phase leakage is a process/editorial failure, not a locked-decision violation — but it still requires correction before handoff.
- The checklist is written to be usable by a human reviewer working alone (author self-review) as well as by a distinct reviewer. The review outcome system does not require a multi-person workflow to be useful.
- Companion files (prompt, answers, review, open-items) are not themselves subject to the checklist's mandatory checks — they are provenance records, not content documents. The checklist covers main documents. CAT-11 checks verify that companion files exist and are correctly named/populated.
- The three-tier model is a reviewer's mental model, not a separate governance document. It is included in the checklist for clarity but does not itself constitute a new policy layer.
