STRUCTURAL DECISIONS
---
- Departed from standard 14-section core doc template; used 18 sections tailored to a system context document
  Rationale: standard sections (e.g., "State / Workflow / Controls") are not the right organizing principle for an ecosystem-level document; the content requires actor-by-actor and boundary-by-boundary structure
- Five-layer ecosystem model used in Section 5: Legal / Platform / Blockchain / Investor-Participant / External Dependency
  Rationale: cleanest way to show what Flamingo controls (Platform Layer) vs. what it depends on (all other layers)
- Four primary boundary definitions in Section 8 (not just one): Platform/Legal, Platform/Blockchain, Platform/Issuer, Platform/Investor
  Rationale: each boundary has distinct authority rules and conflict resolution; conflating them would obscure the architecture
- "Record Truth at Each State" table (Section 10.2) added alongside the interaction flow
  Rationale: shows exactly which system holds truth at each point in the lifecycle without requiring the reader to cross-reference P1-008

---

CONTEXT BOUNDARY DECISIONS
---
- Blockchain layer described as having two components: Base (the network) and ERC-3643 (the token standard)
  Rationale: LD-035 (Base) and LD-036 (ERC-3643) are distinct locked decisions; they serve different functions (execution layer vs. compliance token standard) and should be named separately
- KYC/AML vendor and accreditation provider listed as "pending" in the External Dependency Layer
  Rationale: these are real dependencies for v1 investor intake, but the vendors are not yet selected (UI-008, UI-007 in P1-002); they must appear as ecosystem actors to avoid implying they do not exist
- Operational Registry named as a distinct in-scope "actor/system" in Section 6
  Rationale: it is a system component that requires its own boundary definition; it should not be folded into "Flamingo" without distinguishing the record layer from the workflow engine

---

WHITE-LABEL / TENANT DECISIONS
---
- White-label section (Section 13) structured as three sub-sections: what it is, what it does NOT change (explicit list), and the architecture model
  Rationale: the "does not change" list is the most important content — it prevents the dangerous assumption that white-labeling changes Flamingo's legal role or lifecycle
- Second Street Capital named as launch tenant by name
  Rationale: they are the confirmed v1 tenant per project constraints; naming them makes the document concrete and consistent with P1-004
- Multi-tenant architecture noted as present but full model deferred to Phase 2
  Rationale: the architecture is an established fact (Flamingo is white-label / multi-tenant) but the configuration model detail is Phase 2 scope

---

COMPLETION BOUNDARY DECISIONS
---
- REDEEMED addressed in Section 12 explicitly (not buried in the prohibited assumptions table only)
  Rationale: LD-042 is specific about this; the system context should name it where completion boundaries are defined, not just in a prohibitions list
- TA_ACKNOWLEDGED given a named sub-section note: "not a completion boundary"
  Rationale: this is the most dangerous near-miss in the completion layer — systems might treat TA acknowledgment as close enough to TA recording. Making it explicit at the system context level prevents the confusion from being baked into Phase 3 design.
