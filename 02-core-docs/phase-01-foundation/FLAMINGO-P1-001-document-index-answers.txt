STRUCTURAL DECISIONS
---
- Departed from standard 14-section core doc template; used a 15-section navigation-focused structure
  Rationale: a document index is a map, not a policy or process document; sections like "State/Workflow/Controls" and "Rules and Constraints" do not apply; the index needs sections for each folder tier and audience-specific reading paths

- ASCII directory tree used in Section 4 (Repo Root Structure)
  Rationale: most compact representation of folder hierarchy; readable without tooling; easier to scan than a table

- Phase 1 index uses a full 5-column table (ID / Title / File / Status / Purpose)
  Rationale: Phase 1 documents are DRAFTED — enough detail exists to describe each with a one-line purpose; the fuller table format serves the "new contributor" reading path

- Phase 2 and Phase 3 indexes use a 4-column table (ID / Title / Status / Purpose + gating note)
  Rationale: these are SHELL CREATED; there is no file-name ambiguity; keeping the table narrower avoids listing 32 near-identical filenames

- Workpack pattern described in Section 9 (Operational Support Files) rather than inline in Phase tables
  Rationale: the pattern is the same for all documents; documenting it once avoids 44 rows of repetition; the 5-file structure is already the naming convention

- 5 reading paths provided (New Contributor / Reviewer / Implementation Planner / Phase 2 Drafter / Quick Orientation for Claude Code restart)
  Rationale: Claude Code restart path is unusual for a document index but is highly practical for this project; the repo is large enough that a cold-start context reload benefits from a prescribed reading order
